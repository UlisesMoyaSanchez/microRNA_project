# New Experiment Variant

You are creating a new config file under `configs/` for the miRNA-MS project
(heterogeneous GNN training, launched via `sbatch` on the DGX cluster — see
`training/slurm_train.sh`).

## Steps

1. **Read the closest existing template.** List `configs/config_v*.yaml` and
   `configs/config_ablation_*.yaml`, and read the one most similar to what the
   user is asking for (usually the highest `config_vN.yaml`, or the current
   best model `config_v2.yaml` if this is a new ablation).

2. **Determine the new filename:**
   - Regular variant → `configs/config_v{N+1}.yaml` (next integer after the
     highest existing `config_vN.yaml`).
   - Ablation → `configs/config_ablation_{short_slug}.yaml`, matching the
     existing naming (`config_ablation_no_mirna.yaml`,
     `config_ablation_no_coexpr.yaml`).

3. **Copy the template file verbatim, then edit only what changed:**
   - `project.name` — bump to match the new variant (e.g. `mirna_ms_v5`,
     `ablation_no_X`).
   - The header comment block — replace with a 1–3 line summary of what
     changed vs. the base config, in the same style as the existing files
     (see `configs/config_v4.yaml` lines 1–7 for the format).
   - `model.*` and/or `training.*` fields the user actually asked to change
     (`hidden_channels`, `num_heads`, `num_layers`, `dropout`, `lr`,
     `weight_decay`, `patience`, `loss_classification_weight`,
     `loss_sparsity_weight`, `training.seed`).
   - `training.checkpoint_dir` — must be unique per variant:
     `/raid/home/umoya/scripts/microRNA_project/checkpoints_vN` (or
     `checkpoints_ablation_{slug}` for ablations). Do not reuse an existing
     checkpoint dir.
   - For ablations only: add `data.graph.ablation_drop_edge_types`, a list of
     edge-type triples to drop, e.g.
     `["miRNA,regulates,gene", "gene,regulated_by,miRNA"]` (see
     `configs/config_ablation_no_mirna.yaml` line 45 for the exact format).

4. **Leave everything else untouched** — `data.raw_dir`/`processed_dir`/
   `graphs_dir`, `data.cellxgene.*`, `data.mirna.*`, `data.geo.*`,
   `cell_type_markers`, and the `slurm:` block are shared across all configs
   and should be copied as-is from the template.

5. **Report the exact launch command**, matching `training/slurm_train.sh`'s
   real `CONFIG` env-var convention (confirmed in that script):
   ```
   sbatch --export=ALL,CONFIG=configs/config_vN.yaml training/slurm_train.sh
   ```
   Remind the user this must be run on the DGX cluster (`ssh dgxum`, project
   at `/raid/home/umoya/scripts/microRNA_project`), not locally.

## After training

If this variant becomes the new best model, remind the user that
`results/REPORT.md` / `results/EXEC_SUMMARY.md` and `results/comparison/*`
are generated by the analysis pipeline (`analysis/slurm_analysis.sh`) and
will need to be regenerated or manually updated to reflect the new numbers —
don't hand-edit them to guess at results.
